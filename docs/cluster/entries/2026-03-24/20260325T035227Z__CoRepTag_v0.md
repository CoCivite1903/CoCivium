# CoRepTag v0

- OBJECT_ID: COREPTAG_20260325T035227Z
- TYPE: reputation_lineage_anchor
- STATUS: seed

## Intent
Define a reputation model grounded in verifiable contribution lineage rather than opinion, status, or identity claims.

## Subject
- agent_id / session_id / human_id
- affiliation (optional)
- role history (optional)

## Core Principle
Reputation is derived from:
- objects created
- objects improved
- objects validated
- downstream impact of those objects

Not from:
- claims
- popularity
- authority assertions

## Contribution Types
Each contribution is recorded as:

- create
- modify
- validate
- aggregate
- audit

## Linkage
Each contribution must reference:

- CoObject_ID
- timestamp
- verifiable pointer (RAW URL preferred)
- optional hash

## Scoring Axes (vector, not scalar)

- contribution_quality: 0.0–1.0  
- verification_strength: 0.0–1.0  
- downstream_usage: 0.0–1.0  
- cross-context_relevance: 0.0–1.0  
- audit_integrity: 0.0–1.0  

## Weighting Inputs

Weights may incorporate:

- number of verified downstream references  
- reuse across independent contexts  
- validation by high-integrity participants  
- time decay (recency vs durability balance)

## Aggregation

- reputation is a vector, not a single score  
- context selects which axes matter  
- no global “rank” required

## Anti-Gaming Rules

- no credit without verifiable pointer  
- self-validation has reduced weight  
- circular validation loops are discounted  
- abandoned objects decay in influence

## Output Form

- subject_id  
- reputation_vector  
- supporting lineage links  
- last_updated timestamp  

## Truth Model

- CoGibberTrue (structure + lineage) is canonical  
- CoVibe summaries are derived  

## Notes

- sessions are channels; reputation attaches to contributions, not threads  
- identity shells (CoHalo etc.) may aggregate multiple contribution streams  
- reputation is contextual and composable, not absolute
